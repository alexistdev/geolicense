Bank Transfer Flow (Manual)

USER
  ↓
GET /payment-methods
  → returns PAYMENT_METHOD list
  → BANK_TRANSFER includes all active
    BANK_ACCOUNT rows

USER picks a bank account
  ↓
POST /invoices/{id}/submit-payment
  { paymentMethodId, bankAccountId,
    providerReference }
  ↓
PAYMENT saved
  provider         ← snapshot from BANK_ACCOUNT
  bank_account_id  ← FK
  snapshot_*       ← copied at submit time
  status           = PENDING
  ↓
INVOICE → AWAITING_VERIFICATION
  ↓
ADMIN reviews → approve / reject
  ↓
LICENSE GENERATED

Xendit Flow (Automated)

USER
  ↓
GET /payment-methods
  → returns XENDIT option

USER selects Xendit
  ↓

  { paymentMethodId }
  ↓

  (using PAYMENT_GATEWAY_CONFIG.api_key)

  ↓

Entity Notes

PAYMENT_METHOD New

Admin-created top-level payment option. The type enum controls which child config is used.

  • BANK_TRANSFER → has child BANK_ACCOUNT rows
  • XENDIT → has one PAYMENT_GATEWAY_CONFIG row
  • OTHER → extensible for future gateways
  • sort_order controls display order to users

BANK_ACCOUNT New

One-to-many under a BANK_TRANSFER payment method. Admin adds multiple accounts; user picks which they transferred to.

  • is_main = highlighted/default in the UI
  • is_active = soft toggle without deleting
  • Soft-delete so Payment FK remains valid even after deactivation

PAYMENT_GATEWAY_CONFIG New

Stores provider-specific secrets. One row per gateway-type PaymentMethod (unique FK).

  • api_key — store encrypted at rest
  • webhook_token — used to verify inbound webhook calls
  • extra_config — JSON for provider-specific fields (e.g. Xendit channel type)

PAYMENT Modified

Existing entity — two new FKs and three snapshot fields added.

  • payment_method_id FK — which method was used
  • bank_account_id FK — nullable; only set for BANK_TRANSFER
  • snapshot_bank_name/account_number/account_holder — copied at submit time so the record is self-contained even if the bank account is later deleted
